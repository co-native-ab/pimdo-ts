# ADR-0012: Preview Site — One Generator, HTML + Text + Index, Five-Scenario Matrix

## Status

**Accepted**

## Context

pimdo-ts has three groups of user-visible artefacts:

1. **Browser pages** rendered by `src/templates/*` and served by the
   loopback flows in `src/browser/flows/*` (login, logout, requester,
   approver, confirmer).
2. **MCP `*_list` tool output** — text returned to the AI host for
   every read tool (`pim_group_eligible_list`, …); produced by
   `src/tools/pim/*/format.ts`.
3. **Screenshots** of those pages, useful for at-a-glance visual review
   in PRs.

Before this ADR there was a single thin script,
`scripts/preview-pages.mjs`, that generated only the login/logout pages
under `html-preview/`. The MCP tool output had no preview at all, the
multi-row flows (requester/approver/confirmer) were impossible to eyeball
without running the live server, and there was no way for a reviewer
(or AI agent) to confirm "every list-producing surface still looks
right across an empty / single / pair / full / next-page page".

This created three concrete problems:

- **No diffable artefact** for templates or formatters during code
  review. Changes to `src/templates/row-form.ts` or
  `src/tools/pim/group/format.ts` required manual reproduction.
- **No place to test by browsing.** Contributors had to start the MCP
  server, sign in, and trigger a flow to see what a page looks like.
- **No common "five-scenario" expectation.** Each surface defined its
  own ad-hoc visual tests, if any.

## Decision

Introduce a **single static preview site** at `docs/preview/`,
generated by **one** module under `scripts/preview/`, and verified by
**one** check (`npm run preview:check`).

### Structure

```
scripts/preview/
  scenarios.ts                canonical list-scenario ids
  views.ts                    registry of browser views
  tools.ts                    registry of MCP *_list tools
  fixtures/                   deterministic Graph/ARM/row-form data
  render.ts                   thin wrapper around production templates / formatters
  index-template.ts           inline HTML/CSS/JS for the index page
  generate.ts                 writes docs/preview/
  check.ts                    verifies registration + drift
```

```
docs/preview/
  index.html                  browsable gallery (sidebar + tabs)
  styles.css                  small site CSS
  manifest.json               machine-readable index
  views/<name>/<scenario>/    light.html, dark.html, meta.json
  tools/<name>/<scenario>.{md,html}
```

### Rules

- **No duplication.** `views.ts` and `tools.ts` import directly from
  `src/templates/*` and `src/tools/pim/*/format.ts`. The preview is
  guaranteed byte-identical to what the live MCP server would render
  because it is _the same code_.
- **Five-scenario matrix.** Every list-producing surface ships one
  rendering for each id in `LIST_SCENARIO_IDS = ["empty", "single",
"pair", "full", "next-page"]`.
- **Both colour schemes.** Every browser view ships `light.html` and
  `dark.html` via a `<meta name="color-scheme">` shim — the underlying
  template is rendered once and themed deterministically.
- **Determinism.** Fixed display names ("Alice Example"), sequential
  GUIDs, pinned ISO timestamps (2030-01-01T00:00:00Z), pinned CSP
  nonce / CSRF token. Output is byte-stable across machines.
- **One generator, one check.** `npm run preview` writes the artefacts;
  `npm run preview:check` (in `npm run check` and CI) re-derives the
  expected file contents and refuses any drift. The check also walks
  `src/browser/flows/` and `src/tools/pim/**/*-list.ts` and fails when
  a surface is not registered.
- **Self-explaining failures.** Every failure prints the offending
  surface, the fix steps, ADR-0012, and the `preview-coverage` agent.
  In CI the same nudge is appended to the GitHub Actions step summary.

### Index page

The index (`docs/preview/index.html`) is plain HTML/CSS/JS — no
frameworks. It groups entries by surface family in the sidebar, offers
three tabs per entry (Rendered iframe / Screenshot / Source), supports
a global scenario filter, and toggles light/dark by swapping the iframe
source. It is itself part of the verified output, so renames and
deletions are caught by the same `git diff` style check.

## Consequences

### Positive

- A **single browsable artefact** that exercises every template and
  every list formatter on every canonical scenario.
- Reviewers can open `docs/preview/index.html` locally (or rely on the
  diff in PRs) instead of running the MCP server.
- AI agents have a deterministic loop to close: fix the source, run
  `npm run preview`, run `npm run preview:check`, commit.
- Failures point at the exact ADR + agent that solves them, so neither
  humans nor AI agents need to spelunk.

### Negative

- The repository now ships generated HTML and markdown under
  `docs/preview/`. They are small (no images), deterministic, and
  diff-friendly, but they do grow with each new surface.
- Contributors must remember to run `npm run preview` after changing a
  template or formatter. The check enforces this and the structured
  nudge tells them exactly which command to run.

### Deferred

- **PNG screenshots.** Byte-deterministic PNGs across operating
  systems require an extra image diff tool, so they are explicitly
  out of scope here. The index has a "Screenshot" tab that gracefully
  degrades when no PNG is present, and the framework leaves room for a
  future opt-in `npm run preview:screenshots` job (Playwright) that
  fills in the PNG slots without changing the rest of the architecture.

## Alternatives considered

- **Storybook.** Heavy JS framework dependency, not aligned with
  pimdo's "no UI runtime" stance.
- **Screenshots-only.** Loses interactivity (can't click through, can't
  resize, can't grep the source).
- **HTML-only.** Loses the at-a-glance diffable visual snapshot for
  larger redesigns; mitigated here by keeping the door open for the
  optional screenshot job.
