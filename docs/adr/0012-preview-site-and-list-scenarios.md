# ADR-0012: Preview Site — One Generator, Published to GitHub Pages, Five-Scenario Matrix

## Status

**Accepted**

## Context

pimdo-ts has three groups of user-visible artefacts:

1. **Browser pages** rendered by `src/templates/*` and served by the
   loopback flows in `src/browser/flows/*` (login, logout, requester,
   approver, confirmer).
2. **MCP `*_list` tool output** — text returned to the AI host for
   every read tool (`pim_group_eligible_list`, …); produced by
   `src/tools/pim/*/format.ts`.
3. **Screenshots** of those pages, useful for at-a-glance visual review.

Before this ADR there was a single thin script,
`scripts/preview-pages.mjs`, that generated only the login/logout pages
under `html-preview/`. The MCP tool output had no preview at all, the
multi-row flows (requester/approver/confirmer) were impossible to
eyeball without running the live server, and there was no way for a
reviewer (or AI agent) to confirm "every list-producing surface still
looks right across an empty / single / pair / full / next-page page".

We considered committing the generated artefacts under `docs/preview/`
so they show up in PR diffs, but with 5 views × 2 themes + 12 tools × 5
scenarios that is ~180 generated files churning on every template
tweak — git history quickly becomes unmaintainable. Instead we treat
the preview as a **published artifact**: deterministic locally, never
committed, hosted on GitHub Pages.

## Decision

Introduce a **single static preview site** generated by **one** module
under `scripts/preview/`, written to a gitignored output directory,
and published to GitHub Pages on a per-target subdirectory:

| Trigger             | Pages path          |
| ------------------- | ------------------- |
| push to `main`      | `/preview/main/`    |
| pull request `<id>` | `/preview/pr/<id>/` |

The full URL is `https://<owner>.github.io/<repo>/preview/...`.

### Source layout

```
scripts/preview/
  scenarios.ts                canonical list-scenario ids
  views.ts                    registry of browser views
  tools.ts                    registry of MCP *_list tools
  fixtures/                   deterministic Graph/ARM/row-form data
  render.ts                   thin wrapper around production templates / formatters
  index-template.ts           inline HTML/CSS/JS for the index page
  generate.ts                 writes the output directory
  check.ts                    verifies registration + source coverage
```

### Generated layout (in `.preview/` locally, mirrored on Pages)

```
index.html                  browsable gallery (sidebar + tabs)
styles.css                  small site CSS
manifest.json               machine-readable index
views/<name>/<scenario>/    light.html, dark.html, meta.json
```

> MCP `*_list` tools intentionally do NOT emit preview artefacts —
> they return plain text to the AI by contract, so there's nothing
> meaningful to render in a browser-facing site. Their registration
> coverage is still validated by `preview:check`.

### Rules

- **No duplication.** `views.ts` and `tools.ts` import directly from
  `src/templates/*` and `src/tools/pim/*/format.ts`. The preview is
  guaranteed byte-identical to what the live MCP server would render
  because it is _the same code_.
- **Five-scenario matrix.** Every list-producing surface ships one
  rendering for each id in `LIST_SCENARIO_IDS = ["empty", "single",
"pair", "full", "next-page"]`.
- **Both colour schemes.** Every browser view ships `light.html` and
  `dark.html` via a `<meta name="color-scheme">` shim; the underlying
  template is rendered once and themed deterministically.
- **Determinism.** Fixed display names, sequential GUIDs, pinned ISO
  timestamps (2030-01-01T00:00:00Z), pinned CSP nonce / CSRF token.
- **Never committed.** `.preview/` is gitignored; the generator output
  is rebuilt on every CI run and on every local invocation.
- **One generator, one check.** `npm run preview` writes
  `.preview/`; `npm run preview:check` (in `npm run check` and CI)
  verifies that every browser flow (`src/browser/flows/`) and every
  MCP `*_list` tool (`src/tools/pim/**/*-list.ts`) is registered, and
  that every registered surface renders.
- **Self-explaining failures.** Every check failure prints the
  offending surface, the fix steps, this ADR, and the
  `preview-coverage` agent. In CI the same nudge is appended to
  `$GITHUB_STEP_SUMMARY`.

### CI / publishing

- `.github/workflows/preview.yml` (event: `push:main`,
  `pull_request:main`) builds the site with `contents: read` and
  uploads it as an artifact.
- `.github/workflows/preview-publish.yml` (event: `workflow_run`
  completion of `Preview`) downloads the artifact and pushes it to
  the `gh-pages` branch under the appropriate subdirectory. Splitting
  build and publish keeps untrusted PR code in a permission-less
  context — the elevated `contents: write` token only runs from the
  default-branch workflow definition.
- `.github/workflows/preview-cleanup.yml` (event: `pull_request: closed`)
  removes `preview/pr/<id>/` from `gh-pages` on PR close/merge and
  updates the sticky PR comment.

A sticky PR comment (`marocchino/sticky-pull-request-comment`) carries
the preview URL so reviewers click straight from the PR page.

### Index page

`index.html` is plain HTML/CSS/JS — no frameworks. It groups entries
by surface family in the sidebar, offers three tabs per entry
(Rendered iframe / Screenshot / Source), supports a global scenario
filter, and toggles light/dark by swapping the iframe source. The
Screenshot tab gracefully degrades when no PNG is present (PNG
rendering is an explicit future opt-in — see "Deferred" below).

## Consequences

### Positive

- A **single browsable artefact** that exercises every template and
  every list formatter on every canonical scenario.
- Reviewers click a link in the PR description; they don't need to
  pull, install, or run anything.
- Git history stays clean: only `scripts/preview/**` source and
  fixtures are versioned.
- AI agents have a deterministic loop to close: edit the source, run
  `npm run preview`, open `.preview/index.html`, run
  `npm run preview:check`. CI re-runs the same loop and publishes.
- Failures point at the exact ADR + agent that solves them, so neither
  humans nor AI agents need to spelunk.

### Negative

- The repository now needs a `gh-pages` branch and Pages enabled.
- Forks cannot publish (their PRs run with `contents: read` only); the
  build job still runs and `preview:check` still gates the PR, so
  this only affects the convenience of the published URL.
- Maintainers occasionally need to prune the `gh-pages` branch if it
  grows too large; the cleanup workflow handles per-PR directories
  automatically on close.

### Deferred

- **PNG screenshots.** Byte-deterministic PNGs across operating
  systems require an extra image-diff tool, so they are explicitly
  out of scope. The index has a "Screenshot" tab that gracefully
  degrades when no PNG is present, and the framework leaves room for
  a future opt-in `npm run preview:screenshots` job (Playwright) that
  fills in the PNG slots without changing the rest of the
  architecture.

## Alternatives considered

- **Commit the generated artefacts** under `docs/preview/`. Rejected:
  ~180 churning files per change makes git history unmaintainable
  (this was the initial draft of this ADR).
- **Storybook.** Heavy JS framework dependency, not aligned with
  pimdo's "no UI runtime" stance.
- **Screenshots-only.** Loses interactivity and source-grep.
- **HTML-only, opened locally.** No shareable link; reviewers must
  pull and run.
