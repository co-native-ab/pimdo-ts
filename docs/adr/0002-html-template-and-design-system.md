## Context

pimdo-ts is a stdio-based MCP server distributed as an MCPB bundle with four runtime dependencies (`@modelcontextprotocol/sdk`, `zod`, `@azure/msal-node`, `open`). It serves HTML locally for transient authentication and confirmation flows via loopback HTTP servers — specifically the MSAL login flow (`src/browser/flows/login.ts`) and the row-form flows (`src/browser/flows/requester.ts`, `approver.ts`, `confirmer.ts`).

Currently, HTML templates live in `src/templates/` as TypeScript functions that return HTML strings. However, several problems exist in the current implementation:

- **Duplicated CSS**: the picker template (`src/templates/picker.ts`) duplicates the base styles inline instead of reusing the shared `BASE_STYLE` constant from `src/templates/styles.ts`.
- **Hardcoded colors**: Microsoft-branded blue (`#0078d4`, `#106ebe`, `#005a9e`) and other colors are scattered as magic strings across template files with no single source of truth.
- **No brand consistency**: colors, typography, and spacing have no formal specification — each template uses ad-hoc values.
- **No favicon or icon**: served pages have no visual identity in the browser tab.
- **System fonts only**: the current font stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`) has no relationship to the project's brand typography.

This ADR establishes standing guidelines for how all HTML served by pimdo-ts should be structured. It is a policy document — not a refactoring ADR. All future HTML templates and modifications to existing templates must conform to these guidelines.

Key forces at play:

- **Zero-dependency constraint**: pimdo-ts ships as a self-contained MCPB bundle with three runtime dependencies. Adding template engines, frontend frameworks, or build tools is not acceptable.
- **Transient HTML only**: HTML is served locally on loopback for brief user interactions (sign-in, list selection). These are not web applications — they are momentary browser pages that the user sees for seconds.
- **Testability**: template functions must be unit-testable without a browser or DOM environment.
- **Brand identity**: served pages should have a consistent, recognizable appearance using the project's color palette, typography, and icon — without displaying company names or branding text (the tool is open source).
- **Offline resilience**: the MCPB bundle must function without external network requests for page assets. External font CDNs are acceptable only with graceful fallback.

## Decision

All HTML served by pimdo-ts follows these standing guidelines:

### 1. Pure TypeScript Template Functions

All HTML is generated by TypeScript functions that return `string`. No template engines (Handlebars, EJS, Pug, etc.), no frontend frameworks (React, Vue, Svelte, etc.), and no additional build steps are permitted. Template functions live in `src/templates/` and are imported by server logic files (`src/loopback.ts`, `src/picker.ts`, etc.).

Rationale: TypeScript functions are testable with standard unit test tools, type-safe via the compiler, and add zero runtime dependencies. For transient loopback pages that the user sees for seconds, a full template engine or framework is unjustifiable overhead.

### 2. Design Tokens in `tokens.ts`

A single `src/templates/tokens.ts` file defines all visual constants as typed TypeScript values. Every color, font, spacing, and border-radius value used in served HTML originates from this file. No color values, font names, or spacing values appear as magic strings in template or style files.

The token palette:

**Primary Purple Scale**
| Token | Hex | Usage |
|---------------------|-----------|--------------------------------|
| Brand Purple | `#6659a7` | Primary brand color, buttons |
| Purple +1 | `#3c2f7f` | Hover states, emphasis |
| Purple +2 | `#120457` | Deepest purple, active states |
| Purple −1 | `#9289c0` | Lighter tint, secondary text |
| Purple −2 | `#beb8da` | Subtle backgrounds, borders |
| Purple −3 | `#eae8f3` | Page backgrounds |

**Greyscale**
| Token | Hex | Usage |
|-------------|-----------|--------------------------------------|
| Black | `#000000` | Headings, high-contrast text |
| Grey 700 | `#636466` | Body text |
| Grey 500 | `#8a8c8e` | Secondary text, placeholders |
| Grey 300 | `#c7c8ca` | Borders, dividers |
| Grey 100 | `#efeff0` | Subtle backgrounds |
| White | `#ffffff` | Card backgrounds, button text |

**Complementary (functional use only — success, error, warning — NOT decorative)**
| Token | Primary | Light | Usage |
|----------------|-------------|-------------|--------------------|
| Sandstone | `#dac48a` | `#F8F2E6` | Warning states |
| Teal | `#AABDB5` | `#dae2df` | Success states |
| Cobalt | `#5271AC` | `#DCE2EE` | Info states |
| Peach | `#F9AA8F` | `#FDE4DC` | Error states |

**Typography**: Lexend (Google Fonts), fallback Arial, then sans-serif.

### 3. Shared Styles from a Single Source of Truth

One CSS generation point — `src/templates/styles.ts` — builds all CSS from the tokens defined in `tokens.ts`. Template functions import shared style constants or functions from `styles.ts`. No template file contains inline `<style>` blocks with raw CSS values — all styling flows through the shared module.

This eliminates the current problem where `picker.ts` duplicates the base CSS inline while `login.ts` imports `BASE_STYLE` from `styles.ts`, and both hardcode different color values.

### 4. Font Loading with Graceful Degradation

Lexend is loaded from Google Fonts CDN (`fonts.googleapis.com`) in the `<head>` of served pages. The CSS `font-family` declaration specifies `'Lexend', Arial, sans-serif` so that if the Google Fonts CDN is unreachable (offline, firewalled, or slow), the page degrades gracefully to Arial with no visible breakage or layout shift.

No `font-display: block` or other strategies that would delay rendering while waiting for fonts. The page must render immediately with the fallback font if the web font is unavailable.

### 5. Favicon and Icon as Base64 Data URIs

The SVG icon assets from the `assets/` directory (`symbol.svg`, `symbol-light.svg`, `symbol-dark.svg` for favicons; `logo.svg`, `logo-light.svg`, `logo-dark.svg` for page icons) are embedded in served HTML as base64-encoded SVG data URIs. The favicon uses the symbol (compact mark) and page icons use the logo (full wordmark). No external image requests are made — the MCPB bundle is self-contained.

The base64 encoding is performed by `scripts/encode-icons.mjs`, which generates `src/templates/icons.ts` from the SVG source files. The favicon gives served pages a recognizable identity in the browser tab without requiring any network request.

### 6. XSS Escaping for All Dynamic Values

All dynamic values interpolated into HTML strings must pass through the `escapeHtml()` function (currently in `src/templates/styles.ts`). This applies to URLs, user-provided labels, error messages, and any other value that originates outside the template. No dynamic value is ever interpolated raw into an HTML string.

### 7. What NOT to Do

The following patterns are prohibited in all current and future HTML-serving code:

- **No inline HTML in server logic files**: `src/loopback.ts`, `src/picker.ts`, and similar files must not contain HTML strings. They import template functions from `src/templates/`.
- **No duplicated CSS across templates**: all CSS originates from `src/templates/styles.ts` which reads from `tokens.ts`. Templates do not define their own `<style>` blocks with raw values.
- **No hardcoded color values outside `tokens.ts`**: every hex color, RGB value, or opacity value in served HTML traces back to a named token.
- **No company name or branding text**: the tool is open source. Served pages do not display "Co-native" or any company name. Brand identity is expressed solely through the color palette, typography, and icon.
- **No template engines or frontend frameworks**: no Handlebars, EJS, Pug, React, Vue, Svelte, or similar. Only TypeScript functions returning strings.
- **No external image requests**: all images and icons are embedded as base64 data URIs. The MCPB bundle must render correctly with no network access beyond Google Fonts (which degrades gracefully).

### 7. Row-Form Layout Pattern

Multi-row decision pages — `requesterFlow`, `approverFlow`, and `confirmerFlow` — diverge from the centred login/logout/picker shell on purpose. They are **high-density, console-inspired tables**, not marketing cards: a reviewer triaging twenty pending requests benefits from compact rows, sticky column headers, and tabular metadata far more than from generous whitespace. This pattern is informed by Microsoft Fluent 2 (DataGrid + CommandBar), Atlassian (TableTree + Inline Edit), GitHub Primer (ActionList), Shopify Polaris (ResourceList), and IBM Carbon (DataTable + batch actions bar).

The canonical structure is:

- **Wide left-aligned card.** Override the centred 400px container with a left-aligned `max-width: 1080px` card and reduced padding (`18px 22px 14px`). The body itself flows to `align-items: flex-start` so a long table doesn't try to vertically centre.
- **Slim CommandBar above the table.** A monospace uppercase `BULK` label, a row of borderless text-buttons (e.g. "Approve all", "Include none"), and an aria-live `.row-summary` count on the right (e.g. `3 approve · 0 deny · 2 skip`). Background: `grey1`, border: `grey2`, no rounded corners larger than `borderRadius.sm`.
- **Real `<table class="row-table">` markup** wrapped in a `.row-table-wrap` with `max-height: calc(100vh - 240px)` and `overflow: auto`. Use semantic `<thead>` / `<tbody>` / `<tr>` / `<td>` so screen readers and keyboards get table navigation for free.
- **Sticky uppercase monospace `<thead>`.** Column titles are 0.7rem, `grey3`, `letter-spacing: 0.08em`, `text-transform: uppercase`, `position: sticky; top: 0;` so they remain visible while scrolling long lists.
- **Compact `<tr class="row">`.** Approximately 30–36px tall: cell padding `6px 10px`, line-height `1.45`, `font-size: sm`. Hover row tint: `purple.minus3` (light) / `dark.surfaceHover` (dark). 1px `grey1` separators between rows, no per-row border, no border-radius.
- **Monospace metadata cells.** IDs (subtitle), durations, "max" hints, and the bulk-toolbar `BULK` label use `fontFamilyMono` (the system monospace stack — no extra font load). Names use Lexend; quoted requestor text is italic Lexend in `grey3`.
- **Ultra-compact segmented decision control.** Inside the decision `<td>`, `.decision-group` is a connected segmented control with `border-radius: sm`, ~22px tall, `font-family: fontFamilyMono`, uppercase APPROVE/DENY/SKIP labels at `fontSize.xs`. Selected segment fills with `purple.minus3` and switches text to `purple.brand`. Native radios are visually hidden (`clip: rect(0,0,0,0)`) but kept for keyboard a11y.
- **Inline single-cell controls.** Justification: a 2-row `<textarea class="justification">` with `min-height: 26px` and `max-height: 120px` so it grows on focus / long input. Duration: a `.duration-group` (60px number + flex-1 unit select) capped at 160px with a monospace "max NN h" hint below. Include: a centred 15px native checkbox (`accent-color: purple.brand`).
- **Sticky form-actions footer** with a token-derived white-to-transparent gradient backdrop so Cancel/Submit stay reachable on long lists.
- **Skipped state via typography, not opacity.** Strikethrough `.row-label` plus dimmed `.cell-meta` / `.cell-quote`. Applied only after explicit user action (radio change to Skip or bulk Skip-all) — never on initial paint, even if "Skip" is the default radio.
- **Empty state in console voice.** A single muted monospace line such as `// no pending approvals`, no card chrome.

These rules apply to every new row-form surface added in `src/browser/flows/`. Reuse `ROW_FORM_STYLE` and the helpers in `src/templates/row-form.ts` rather than authoring per-flow CSS, and add `fontFamilyMono` from `tokens.ts` (never an inline font stack).

## Consequences

### Positive

- **POS-001**: A single `tokens.ts` file provides one source of truth for all visual constants — changing a brand color updates every served page automatically with no risk of inconsistency.
- **POS-002**: Template functions are unit-testable with vitest, asserting on HTML structure, token usage, XSS escaping, and favicon presence — no browser or DOM environment needed.
- **POS-003**: Zero new runtime dependencies are introduced. The template approach works entirely within TypeScript's existing capabilities and the project's three-dependency constraint.
- **POS-004**: Base64-embedded favicons make the MCPB bundle fully self-contained for served HTML — no external asset requests that could fail in air-gapped or restricted network environments.
- **POS-005**: The prohibition on company names and branding text keeps served pages appropriate for an open-source project, expressing identity through subtle visual design (colors, typography, icon) rather than proprietary marks.
- **POS-006**: Graceful font degradation to Arial ensures served pages render immediately and correctly regardless of network conditions — no blank text, no layout shifts, no loading delays.

### Negative

- **NEG-001**: TypeScript string templates provide no syntax highlighting, auto-completion, or structural validation for HTML — developers must rely on unit tests and manual review to catch malformed markup.
- **NEG-002**: Base64-encoded favicons increase the HTML payload size (a 16×16 PNG adds ~1–2 KB of base64 text per page), though this is negligible for loopback-served pages.
- **NEG-003**: The prohibition on template engines means complex future pages (if ever needed) will require increasingly elaborate TypeScript string composition — though the transient, simple nature of these pages makes this unlikely to become a problem.
- **NEG-004**: Depending on Google Fonts CDN for Lexend means the branded typography is not guaranteed — users behind strict firewalls will see Arial instead. The page remains functional and readable, but the visual identity is reduced.
- **NEG-005**: Enforcing that all colors flow through `tokens.ts` requires discipline from contributors — a hardcoded hex value in a template passes TypeScript compilation and only fails if caught by code review or linting rules.

## Alternatives Considered

### Template Engine (Handlebars, EJS, Pug)

- **ALT-001**: **Description**: Use a lightweight template engine like Handlebars, EJS, or Pug to generate HTML, providing proper syntax highlighting, structural validation, and a separation of template markup from logic.
- **ALT-002**: **Rejection Reason**: Adds a runtime dependency to a project that currently has three. The MCPB bundle size increases, and the dependency must be maintained, audited for vulnerabilities, and kept compatible. For transient loopback pages that the user sees for seconds, the overhead is unjustifiable. TypeScript string functions achieve the same result with zero added cost.

### Frontend Framework (React SSR, Preact, Svelte)

- **ALT-003**: **Description**: Use a frontend framework in SSR mode (e.g., `renderToString` from React/Preact, or Svelte's SSR compiler) to generate HTML, benefiting from component composition, JSX syntax highlighting, and a rich ecosystem.
- **ALT-004**: **Rejection Reason**: Fundamentally disproportionate for the use case. A framework adds multiple dependencies, a build step (JSX/TSX compilation), and conceptual overhead for pages that contain a heading, a button, and a countdown timer. The component model provides no value when each "page" is a single self-contained function.

### Self-Hosted Fonts (Bundle Lexend in the Package)

- **ALT-005**: **Description**: Bundle the Lexend font files (WOFF2) directly in the MCPB package and serve them as base64 data URIs or from the loopback server, eliminating the dependency on Google Fonts CDN entirely.
- **ALT-006**: **Rejection Reason**: Lexend WOFF2 files add ~50–100 KB to the bundle for a cosmetic improvement that is only visible for the few seconds the user interacts with the page. The Arial fallback is acceptable, and the graceful degradation approach avoids the bundle size cost while still providing the branded typography when the CDN is reachable.

### CSS-in-JS Library (e.g., Emotion, Styled-Components)

- **ALT-007**: **Description**: Use a CSS-in-JS library to manage styles programmatically with full TypeScript integration, automatic scoping, and token-aware style generation.
- **ALT-008**: **Rejection Reason**: Adds a runtime dependency and requires either a build step or runtime CSS injection. The project's styling needs are simple enough that a plain CSS string built from TypeScript token constants achieves the same token-awareness without any added dependencies. CSS-in-JS is designed for complex, interactive web applications — not transient loopback pages.

### No Design Tokens (Continue with Ad-Hoc Colors)

- **ALT-009**: **Description**: Continue the current approach of hardcoding color values directly in template files, accepting the inconsistency and maintenance cost.
- **ALT-010**: **Rejection Reason**: The current state already demonstrates the problem — Microsoft-branded blues (`#0078d4`) are scattered across files with no connection to the project's actual brand palette, and the picker template duplicates CSS that the login template shares. As more HTML pages are added (e.g., future Graph surface configuration flows), the inconsistency compounds. A single `tokens.ts` file is trivial to create and eliminates the class of bugs where a color is updated in one place but missed in another.

## Implementation Notes

- **IMP-001**: Create `src/templates/tokens.ts` exporting typed constants for all color, typography, and spacing values defined in this ADR. Use a flat object or namespace — no nested hierarchies that complicate imports.
- **IMP-002**: Refactor `src/templates/styles.ts` to import all values from `tokens.ts`. Replace `BASE_STYLE` with a function or constant that generates CSS from tokens. Export the `escapeHtml()` utility function unchanged.
- **IMP-003**: Update `src/templates/login.ts` and `src/templates/picker.ts` to import shared styles from `styles.ts` and use no hardcoded color, font, or spacing values. Add `<link rel="icon">` with base64-encoded favicon and Google Fonts `<link>` for Lexend.
- **IMP-004**: Add unit tests for all template functions covering: valid HTML structure (doctype, html/head/body), token usage (brand colors present in output), XSS escaping (dynamic values properly escaped for `<`, `>`, `"`, `&`), and favicon (base64 data URI present in output).
- **IMP-005**: Consider adding an ESLint rule or grep-based CI check that flags hexadecimal color literals in `src/templates/*.ts` files other than `tokens.ts`, to enforce the "no hardcoded colors" policy automatically.
- **IMP-006**: Generate the base64 favicon constant at build time (in `build.mjs`) or as a static constant in `tokens.ts` — do not encode at request time in the HTTP handler.
- **IMP-007**: Success criteria: every served HTML page uses only token-derived colors, loads Lexend with Arial fallback, displays a favicon, escapes all dynamic values, and passes unit tests for structure, tokens, XSS, and favicon.
